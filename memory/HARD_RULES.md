# Hard Rules

> These rules are absolute. Never violate them. Read before every experiment.
> Add new rules when experiments reveal permanent constraints.

1. **n_obs=2 with 64 envs + 600s is insufficient** -- exp_013 only completed 297k/500k steps, reward regressed from 7.36 to 5.02. Need 1024+ envs on GPU. exp_014 proved this: 1024 envs + GPU -> 7.29 reward in 206s.
2. **Level 2 requires adaptive trajectories** -- all controllers (state, PID, RL) use fixed polynomial trajectories from hardcoded waypoints. When gates are randomized on Level 2, the trajectory flies through empty space. 0/5 finishes for every controller.
3. **ONE_D_RPM action space (hover backend) caps reward at ~474** -- tested quartic, quadratic, velocity penalty, conservative PPO. The bottleneck is 1 action dimension, not the reward function.
4. **Don't confuse fast crashes with fast laps** -- exp_013 averaged 4.30s on Level 2 with 0/5 finishes. The low time = early crash, not speed.
5. **Kaggle winners likely use dynamic path planning** that adapts waypoints to observed gate positions, not the static spline trajectory all current controllers use.
6. **Inference-time trajectory swapping doesn't work** -- attitude_rl_dynamic.py built splines from obs["gates_pos"] using the exp_010 policy. Result: 0/5 finishes, 0-1 gates on L2 (4.64s avg = crashes). The policy is tightly coupled to the training trajectory shape. Fix requires training on diverse trajectories, not just swapping at inference.
7. **Training reward plateaus ~7.7 regardless of steps** -- exp_016 (10M steps) only reached 7.71 vs exp_015's 7.53 at 3M. Diminishing returns after ~6M steps. Need architectural changes, not more compute.
8. **GPU training works but RunPod SSH proxy blocks SCP/file transfer** -- use deploy key to git push directly from pod. Deploy key (read-write) is registered on GitHub. Private key is at `/media/id_ed25519_runpod` (shared folder, NOT in the git repo). To set up a pod: copy key to pod via `cat /media/id_ed25519_runpod | ssh pod 'mkdir -p ~/.ssh && cat > ~/.ssh/id_ed25519 && chmod 600 ~/.ssh/id_ed25519'`, then run `setup_runpod.sh`.
9. **exp_016 is the first RL to finish Level 2** -- 13.49s, 2/10 finishes (20%). Better than reference RL (0/5) but 3-4x slower than Kaggle winners (~3.4-5.0s).
10. **Training env (RandTrajEnv) has ZERO gate awareness** -- the RL agent trains on random trajectory following, not gate racing. It never sees gate positions, never gets gate-passage rewards. The only configurable reward params are penalties (rpy, action smoothness, energy). Adding a gate reward requires modifying train_rl.py's RandTrajEnv or building a new training pipeline on RaceCoreEnv.
11. **Obs space is 73 dims: drone state (13) + trajectory lookahead (30) + history (26) + last action (4)** -- no gate info is included. The agent can only follow trajectories, not navigate to gates.
12. **Gate-aware trajectory training works (exp_018)** -- Modified RandTrajEnv.reset() to generate splines through gate positions with L2 randomization. Config flag: `gate_aware: true` in YAML.
13. **RunPod GPU: RTX 3090 (24GB VRAM), $0.47/hr active / $0.01/hr idle** -- Pod name: "desirable_brown_mongoose" (ID: l4lu7w9i2rvfxm). PyTorch template. All GPU experiments (014-016, 019-020) run on this. 1024 envs fits comfortably in 24GB. SSH TCP endpoint supports SCP (use `-p PORT`). Always document GPU type in experiment results.
14. **Gate-aware midpoint trajectories crash at gate 1->2 transition (exp_019/020)** -- All gate-aware models (3M and 10M GPU) crash at ~5.7s after passing gate 1. The climb from z=0.7 (gate 1) to z=1.2 (gate 2) + lateral direction change is too aggressive for the simple midpoint spline. Need approach/departure vectors based on gate yaw, or more intermediate waypoints per gate.
15. **RunPod TCP SSH (exposed port) supports SCP/SFTP** -- Use `ssh -p PORT` with the TCP endpoint, not the proxy. This enables direct file transfer. Much better than base64-over-TTY.
16. **Yaw-aware trajectories fix gate 1->2 crash (exp_021)** -- Using gate quaternion yaw for approach/departure vectors (d=0.5m) passes 2-3 gates on L0 (vs 1 gate with midpoints). Wider distances (0.75m, 1.0m) are worse. Altitude interpolation waypoints don't help. Best L2 result: 3 gates, but 0 finishes and 40% early crash rate.
17. **Trajectory-following approach has hit its ceiling** -- exp_021 is the best trajectory-fix result achievable without retraining. The remaining gap (0 finishes vs competition sub-5s) requires a fundamentally different approach: training directly on RaceCoreEnv with gate-proximity reward, not trajectory following.

20. **Altitude control is SOLVED (exp_026), horizontal navigation is not** -- Adding vz_coef=0.5 + z_high=1.3 + gamma=0.97 produced the first model to hover stably at gate altitude (z≈0.72, gate 0 at z=0.70) for 28.8s. Flight time jumped from 1.16s → 28.8s. But model stays near start position and never navigates horizontally toward gates. Proximity reward alone is insufficient for lateral navigation — needs random gate starts (exp_027) to learn approach/passage behavior.
21. **100% random gate starts skip ground takeoff (exp_027)** -- Training with `random_gate_start: true` + `random_gate_ratio: 1.0` always spawns drone mid-air near a gate. Reward was high (11.67) but benchmark crashed at 1.1s — model never learned to take off from z=0.01. Fix: use `random_gate_ratio: 0.5` (50/50 ground + random gate starts) so model learns both takeoff AND gate approach.
22. **Random gate starts from scratch don't work even at 50/50 (exp_027b)** -- Mid-air random gate envs dominate the reward signal even with only 50% ratio. Model learns max-thrust from mid-air spawns and never learns ground hover. Reward 10.79, benchmark 0 gates, 1.18s. Must fine-tune from a pretrained hover model (exp_026) instead of training from scratch.
23. **Random gate starts destroy hover even when fine-tuning (exp_027c)** -- Fine-tuning exp_026 with 50% random gates + LR=0.0005 partially preserved takeoff (3.2s vs 1.1s scratch) but largely destroyed hover (3.2s vs 28.8s exp_026). The mid-air reward signal erodes ground-start skills regardless of initialization. Random gate starts are incompatible with the current reward structure. Alternative: increase speed_coef to incentivize horizontal navigation without changing spawn logic.
24. **speed_coef=1.0 teaches navigation but destroys hover (exp_028)** -- First-ever RaceCoreEnv gate passage (1 gate in run 4!) but avg flight time 0.94s (vs 28.8s exp_026). speed_coef=1.0 overwhelms hover/altitude rewards (~1.8/step). The sweet spot is 0.3-0.5: enough lateral incentive to move toward gates, not so much that it destabilizes hover. Also use lower LR to preserve pretrained hover skill.
25. **speed_coef has NO navigation sweet spot with exp(-k*dist) proximity** -- Full sweep: 0.4 hover (29.98s), 0.55 edge (25.98s, 1 crash), 0.70 crash (2.02s, 0 gates), 1.0 crash+navigate (0.94s, 0.2 gates). Sharp phase transition between hover and crash with no intermediate regime. The proximity reward exp(-k*dist) is the root cause — it rewards absolute closeness (hovering near gate = free reward forever), not movement toward gate. Only exp_028 (1.0) achieved gates, but only because the extreme speed signal was strong enough to overcome the hover trap despite crashing. Fix: PBRS delta-progress reward (exp_032) — rewards only gate approach, not proximity.
26. **PBRS progress reward breaks hover trap but needs stability (exp_032)** -- progress_coef=50 + speed_coef=0.3 successfully broke the hover local optimum (model moves laterally, 2.92s flight vs 29.98s hover). But crashes before reaching gates. The reward is correct (Ng et al. 1999) — the policy just needs better training to balance approach with stability. Next: truncation fix (exp_033) to improve value estimates near episode boundaries.
27. **Truncation fix restores hover but makes model too conservative (exp_033)** -- Pardo et al. fix (bootstrap through timeouts) improved value estimates dramatically. Flight time jumped from 2.92s (exp_032) to 24.58s (4/5 runs at max 29.98s). But 0 gates — back in hover trap. The improved value function correctly estimates that hovering is safer than navigating, reinforcing the local optimum. PBRS + truncation fix alone (speed_coef=0.3) is insufficient. Need higher speed_coef to create enough lateral incentive to overcome the improved value-based conservatism.
28. **Navigation problem is binary: crash-or-hover with no middle ground** -- exp_026-034 show two regimes: (A) crash+navigate (exp_028/031/032: ≤2.92s, occasional gates) and (B) hover-only (exp_026/029/030/033/034: ≥24.58s, 0 gates). No experiment has achieved stable navigation (>5s flight AND >0 gates).
29. **PBRS eliminates speed phase transition but survive_coef anchors hover (exp_034)** -- speed_coef=0.7 with PBRS: 29.98s stable hover, 0 gates (vs exp_031 with proximity: 2.02s crash, 0 gates). PBRS fixed the crash problem but the deterministic mean policy still hovers. Training reward was high (17.26, peaks 24) from stochastic exploration, but the policy mean never shifts. Root cause: survive_coef=0.5 + alt_coef=1.5 = 2.0/step guaranteed for doing nothing. This is the highest reward-to-risk ratio, so the policy mean stays at hover. Fix: remove survive_coef to make hover less profitable than navigation.
30. **survive_coef brackets: 0.5=hover, 0.0=crash, no gates either way (exp_034/035)** -- survive_coef=0.5 + PBRS + speed=0.7 = 29.98s hover, 0 gates. survive_coef=0.0 + PBRS + speed=0.7 = 0.96s crash, 0 gates. Without survival reward, alt_coef=1.5 alone is insufficient to maintain hover stability — model crashes same as exp_028 (0.94s). The survive parameter has a BINARY phase transition similar to speed_coef. Sweet spot might be 0.1-0.2, or the problem needs curriculum/architecture changes.
31. **Training reward bimodality signals navigation capability (exp_035)** -- Mean reward 11.32 but strongly bimodal (peaks 24, dips 2-5). High peaks = successful navigation attempts during stochastic exploration. Low dips = crashes. The policy IS capable of navigating when exploration noise is favorable, but the learned mean policy converges to crash (survive=0) or hover (survive=0.5). The optimization landscape has competing attractors.
